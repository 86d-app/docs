Modules are the building blocks of an 86d store. Each module is a self-contained npm package under the `@86d-app` scope that ships three things together: API endpoints your storefront calls, admin UI pages that appear in the dashboard, and React components you can drop directly into your MDX templates. You add or remove features by editing a single list in `config.json` — there are no plugins to configure, no monolithic bundles to untangle.

## What a module provides

Every module in the `@86d-app` scope is structured the same way:

- **Store endpoints** — public API routes mounted at `/api/[module-path]/...`
- **Admin endpoints** — protected routes accessible only to authenticated admin users
- **Store components** — React components (e.g. `<Cart />`, `<ProductGrid />`) registered automatically in the MDX component registry
- **Admin components** — management UI pages that appear in the relevant group in the sidebar

Modules depend only on `@86d-app/core` and communicate with each other through declared contracts, not direct imports. This keeps every module independently replaceable.

## Enabling modules in config.json

The `modules` field in `templates/brisa/config.json` controls which modules are active. After editing it, run `bun run generate:modules` to wire everything in.

```json config.json
{
  "modules": [
    "@86d-app/cart",
    "@86d-app/products",
    "@86d-app/checkout",
    "@86d-app/reviews"
  ]
}
```

<Steps>
  <Step title="Edit config.json">
    Add the module package name to the `modules` array in `templates/brisa/config.json`.
  </Step>
  <Step title="Regenerate module imports">
    ```bash
    bun run generate:modules
    ```

    This installs any missing packages, updates `package.json`, and regenerates `modules.ts` with static imports from every enabled module.
  </Step>
  <Step title="Use the components in MDX">
    Components from newly enabled modules are immediately available in any MDX template file — no import statements needed.
  </Step>
</Steps>

<Note>
  Setting `"modules": "*"` in `config.json` enables all installed modules. This is the default for the brisa starter template.
</Note>

## Using module components in MDX

Once a module is enabled, its components are auto-registered in the MDX component registry. You use them like any other JSX element inside your `.mdx` files:

```mdx index.mdx
<FeaturedProducts limit={4} title="Staff Picks" />

<CollectionGrid featured />

<Reviews productId={id} />

<Cart />
```

No import statements are required — the code generator handles that automatically.

## Installing community modules from npm

Modules do not have to come from the `@86d-app` scope. Any npm package that follows the 86d module contract works the same way. Add the package name to `config.json` and run `bun run generate:modules` — the generator detects that it is not a workspace package and installs it from npm automatically.

```json config.json
{
  "modules": [
    "@86d-app/cart",
    "@86d-app/products",
    "@some-company/analytics-widget"
  ]
}
```

<Tip>
  You can enable and disable modules between environments by keeping separate `config.json` files — for example, including a `@86d-app/debug-tools` module in development but omitting it from production.
</Tip>
